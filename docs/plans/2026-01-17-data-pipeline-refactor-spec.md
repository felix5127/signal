# Signal Hunter 数据采集管线重构需求规范

> 文档版本: v1.0
> 创建日期: 2026-01-17
> 状态: 待确认

---

## 一、背景与问题

### 1.1 当前痛点

1. **过滤逻辑不统一**
   - `SignalFilter`（用于 HN/GitHub）要求 A/B/C/D/E/F 条件，过于严格
   - `InitialFilter`（用于 Blog）使用 0-5 评分，相对宽松
   - 两套逻辑导致行为不一致，难以维护

2. **过滤效果差**
   - HN 数据源：110 次采集，保存数全为 0
   - 大量有价值的深度观点文章被误杀
   - 无法观测过滤过程，无法迭代优化

3. **缺乏反馈闭环**
   - 无法知道哪些内容被过滤、为什么被过滤
   - 无法通过人工校验来优化过滤规则
   - Prompt 修改需要改代码重新部署

4. **数据源管理不透明**
   - 不知道每个数据源的采集效果
   - 无法动态调整数据源配置

### 1.2 目标

构建一个**可观测、可迭代、数据驱动**的内容采集管线，通过人机协作持续优化过滤准确率。

## 二、内容定位

### 2.1 内容标准

| 维度 | 标准 |
|------|------|
| 领域 | AI/LLM 技术情报 |
| 深度 | 深度分析优先，非简单新闻通稿 |
| 来源 | 头部公司产品优先（OpenAI/Anthropic/Google/Meta） |
| 影响力 | 能改变开发者工作方式的产品（Cursor, Claude Code 等） |

### 2.2 内容类型优先级

| 优先级 | 类型 | 说明 |
|--------|------|------|
| P0 | 技术文章/博客 | 深度技术分析、经验分享 |
| P1 | Twitter/推文 | AI 领域 KOL 动态 |
| P1 | Podcast/播客 | 深度访谈、行业洞察 |
| P1 | Video/视频 | 技术教程、产品演示 |
| P2 | HN/GitHub/arXiv | 暂缓，后续再优化 |

### 2.3 过滤原则

**分层过滤**：不同类型内容使用不同过滤标准

- 白名单源 → 直接入库，不经过 LLM 过滤
- 其他源 → LLM 评分过滤

## 三、数据源策略

### 3.1 核心数据源（优先实现）

| 数据源 | 采集方式 | 过滤策略 |
|--------|----------|----------|
| Blog | RSS/OPML | 白名单源直接入库，其他 LLM 评分 |
| Twitter | RSS Bridge | 白名单账号直接入库，其他 LLM 评分 |
| Podcast | RSS | 白名单源直接入库 |
| Video | RSS | 白名单源直接入库 |

### 3.2 次要数据源（暂缓）

- Hacker News
- GitHub Trending
- arXiv
- HuggingFace
- ProductHunt

### 3.3 采集配置

| 参数 | 值 | 说明 |
|------|-----|------|
| 采集频率 | 每 6 小时 | 平衡实时性和重复率 |
| 去重策略 | 混合 | URL 精确匹配 + 标题相似度 + 内容指纹 |
| 数据保留 | 永久 | 所有采集内容都保留，包括未通过的 |

### 3.4 语言策略

| 语言 | 处理方式 |
|------|----------|
| 中文 | 保留原文 |
| 英文 | 完整翻译（标题+摘要+全文） |
| 其他 | 不采集 |

## 四、过滤与审核机制

### 4.1 过滤流程

```
┌─────────────┐
│  RSS 采集   │
└──────┬──────┘
       ▼
┌─────────────┐     是      ┌─────────────┐
│ 白名单源？  │ ─────────▶ │  直接入库   │
└──────┬──────┘             │ status=published
       │ 否                 └─────────────┘
       ▼
┌─────────────┐
│ URL 去重    │
└──────┬──────┘
       ▼
┌─────────────┐
│ 全文提取    │
└──────┬──────┘
       ▼
┌─────────────┐
│ LLM 评分    │ ──▶ 保存评分结果和原因
└──────┬──────┘
       │
   ┌───┴───┐
   ▼       ▼
 通过    未通过
   │       │
   ▼       ▼
┌─────┐  ┌─────────┐
│发布 │  │待审核   │
│池   │  │池       │
└─────┘  └─────────┘
```

### 4.2 内容状态机

```
                    ┌──────────────┐
                    │   pending    │ ← 刚采集，待 LLM 评分
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ approved │ │ rejected │ │published │
        │ 待审核   │ │ 被拒绝   │ │ 白名单源 │
        └────┬─────┘ └────┬─────┘ └──────────┘
             │            │
    人工审核 │            │ 人工恢复
             ▼            ▼
        ┌──────────────────────┐
        │      published       │ ← 对外可见
        └──────────────────────┘
```

### 4.3 LLM 评分标准

**统一评分制（0-5 分）**：

| 分数 | 含义 | 处理 |
|------|------|------|
| 0 | 完全不相关/垃圾内容 | rejected |
| 1 | 关联很弱 | rejected |
| 2 | 有一定相关性但价值低 | rejected |
| 3 | AI 相关，有一定深度 | approved（待审核） |
| 4 | 高质量，有价值见解 | approved |
| 5 | 极高质量，强烈推荐 | approved |

**通过阈值**：>= 3 分进入 approved 状态

### 4.4 白名单管理

- **管理方式**：后台界面动态添加/删除
- **白名单类型**：
  - 源白名单（RSS Feed URL）
  - 作者白名单（Twitter 账号、博主）
- **白名单行为**：直接进入 published 状态，跳过 LLM 评分

## 五、后台审核系统

### 5.1 审核界面

**布局**：左右分栏对比

```
┌─────────────────────────────────────────────────────────┐
│                    审核工作台                            │
├───────────────────────┬─────────────────────────────────┤
│   LLM 通过 (approved) │      LLM 拒绝 (rejected)        │
├───────────────────────┼─────────────────────────────────┤
│ ┌───────────────────┐ │ ┌───────────────────┐           │
│ │ 文章卡片          │ │ │ 文章卡片          │           │
│ │ - 标题            │ │ │ - 标题            │           │
│ │ - 来源 / 时间     │ │ │ - 来源 / 时间     │           │
│ │ - LLM 评分: 4     │ │ │ - LLM 评分: 2     │           │
│ │ - LLM 理由: xxx   │ │ │ - LLM 理由: xxx   │           │
│ │                   │ │ │                   │           │
│ │ [删除] [批注]     │ │ │ [恢复] [批注]     │           │
│ └───────────────────┘ │ └───────────────────┘           │
│         ...           │         ...                     │
└───────────────────────┴─────────────────────────────────┘
```

**操作**：

| 操作 | 说明 |
|------|------|
| 删除 | 将 approved 内容改为 rejected + 记录原因 |
| 恢复 | 将 rejected 内容改为 published + 记录原因 |
| 批注 | 添加文字说明，用于 Prompt 优化参考 |

### 5.2 审核工作量

- **目标**：每天约 200 条
- **时间**：约 30-60 分钟
- **目的**：建立标注数据集，迭代优化 Prompt

### 5.3 反馈利用

**半自动模式**：

1. 系统根据人工批注自动分析模式
2. AI 提出 Prompt 优化建议
3. 用户确认后生效

## 六、数据源管理

### 6.1 数据源列表页

显示所有 RSS 源，支持：

- 按类型分类（Blog/Twitter/Podcast/Video）
- 添加/删除数据源
- 启用/禁用数据源
- 设置为白名单

### 6.2 数据源统计

| 指标 | 说明 |
|------|------|
| 采集总量 | 该数据源总共采集了多少条 |
| 通过数量 | LLM 评分 >= 3 的数量 |
| 通过率 | 通过数量 / 采集总量 |
| 人工改判数 | 人工从 rejected 恢复的数量 |
| 误杀率 | 人工改判数 / 拒绝数量 |
| 平均评分 | 该数据源内容的平均 LLM 评分 |
| 最近采集 | 最近一次采集时间 |
| 采集状态 | 正常/异常/已禁用 |

### 6.3 数据源详情

点击数据源可查看：

- 最近采集的内容列表
- 评分分布图表
- 人工改判记录

## 七、Prompt 版本管理

### 7.1 Prompt 存储

```yaml
# prompts/filter_v1.yaml
version: 1
name: "内容过滤 Prompt v1"
created_at: "2026-01-17"
status: active  # active / archived / testing

system_prompt: |
  你是一个 AI 技术内容筛选专家...

user_prompt_template: |
  请评估以下内容：
  标题：{title}
  来源：{source}
  内容：{content}
```

### 7.2 版本功能

| 功能 | 说明 |
|------|------|
| 版本历史 | 查看所有历史版本 |
| 回滚 | 恢复到之前的版本 |
| A/B 测试 | 同时运行两个版本，对比效果 |
| 效果对比 | 对比不同版本的通过率、误杀率 |

## 八、处理流程

### 8.1 内容处理管线

```
采集 → 去重 → 全文提取 → LLM评分 → 摘要生成 → [翻译] → 存储
```

### 8.2 摘要格式

**结构化摘要**（200-300 字）：

```
【背景】一两句话说明文章背景和上下文
【核心观点】文章的主要论点和关键信息
【结论】作者的结论或建议
```

### 8.3 翻译策略

| 字段 | 中文内容 | 英文内容 |
|------|----------|----------|
| 标题 | 保留原文 | 翻译 |
| 摘要 | 保留原文 | 翻译 |
| 全文 | 保留原文 | 翻译 |

## 九、技术约束

### 9.1 LLM 选择

- **优先级**：质量优先，成本次要
- **建议模型**：Claude 3.5 Sonnet / GPT-4o

### 9.2 前端展示

- **保持现状**，本次重构专注后台

### 9.3 交付方式

- **全面重构**，一次性交付

## 十、验收标准

### 10.1 功能验收

- [ ] 后台可查看所有采集内容（通过/拒绝分组）
- [ ] 可对内容进行删除/恢复操作并添加批注
- [ ] 可管理数据源（添加/删除/启用/禁用/白名单）
- [ ] 可查看每个数据源的统计指标
- [ ] 可管理 Prompt 版本（查看/回滚）
- [ ] 采集管线正常运行（6 小时一次）
- [ ] 去重功能正常（URL + 标题 + 内容指纹）
- [ ] 英文内容自动翻译

### 10.2 数据验收

- [ ] 每天采集 200+ 条内容进入审核池
- [ ] 白名单源内容直接发布
- [ ] 所有采集内容都有保留（包括 rejected）
